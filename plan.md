# ProgressiveStages 3.0 Final Plan

## Purpose

ProgressiveStages 3.0 will become a configurable progression platform for Minecraft rather
than a collection of fixed stage checks. Pack authors must be able to define progression,
presentation, visibility, costs, rewards, enforcement, automation, scope, and compatibility
without requesting a new hardcoded feature for every pack design.

This document is the execution contract for the 15 approved feature groups. It separates
the architectural foundation from authoring tools and long term progression modes so that
each phase can ship with migration support, tests, and rollback safety.

## Current status

The 3.0 final baseline has completed its automated Phase A audit, regression tests, clean build,
artifact inspection, configuration migration, merged trigger, UI, command, KubeJS, and optional
integration checks. The real client, dedicated server, multiplayer, and optional mod runtime
matrix remains the final manual Phase A release gate.

Phases B through F are the approved development roadmap. Their registry kernel, universal
condition tree, transaction engine, policy and action registries, advanced progression modes,
Author Studio, pack tooling, journal, and performance framework are not marked complete until
their individual exit gates and verification matrix pass.

## Product rules

1. Configuration is data first. Core behavior is selected by registered identifiers and
   configured values instead of Java enums or parser switch statements wherever practical.
2. Every server mutation is authoritative, validated, atomic, observable, and auditable.
3. Conditions are event driven. Arbitrary expressions are never evaluated every tick.
4. The player UI follows vanilla advancement screen behavior and accessibility conventions.
5. The authoring UI writes the same public format that can be edited by hand or generated by
   scripts. There is no hidden second configuration model.
6. Optional integrations use service interfaces and compatibility modules. Missing mods must
   never break core startup.
7. Existing 2.x and early 3.0 configurations receive deterministic migrations with backups.
8. Multiplayer semantics are explicit. Every rule declares player, team, or server scope.
9. FTB Quests remains the quest and narrative system. ProgressiveStages expands its existing
   stage provider, conditions, rewards, diagnostics, and synchronization bridge.
10. Personalized world generation is out of scope. Shared world enforcement uses guarded
    drops, interaction policies, and client visual substitution.

## Configuration target

```text
config/
└── progressivestages/
    ├── progressivestages.toml
    ├── variables.toml
    ├── templates/
    ├── profiles/
    ├── history/
    ├── backups/
    └── stages/
        ├── early_game.toml
        └── technology/
            └── electricity.toml
```

Stage files remain the owner of their triggers, conditions, display information, costs,
rewards, choices, and enforcement policies. Reusable definitions may be referenced from
templates and profiles, but a loaded stage always resolves to one complete immutable model.

## The 15 feature groups

### 1. Universal condition engine

Replace trigger specific boolean lists with a reusable condition abstract syntax tree.

Required nodes:

- `all`, `any`, `not`, and `at_least` composition.
- Registered leaf predicates with namespaced identifiers.
- Numeric and string comparison operators.
- Inclusive and exclusive ranges.
- References to named reusable predicates.
- Optional conditions that warn when their provider is unavailable.
- A human readable explanation for every failed node.

The engine is shared by unlock, visibility, purchase, revoke, enforcement, reward, command,
and UI rules. Existing `[[triggers]]` entries compile into the new tree and remain supported.

Execution model:

1. Parse configuration into an unbound tree.
2. Resolve predicate identifiers through the condition registry.
3. Validate provider configuration with its codec and schema.
4. Compile the tree into an immutable executable plan.
5. Index the plan by the events and state keys it consumes.
6. Reevaluate only affected plans when an indexed event arrives.

Completion gates:

- Nested truth table tests and malformed tree tests.
- Dependency and cycle validation for named predicate references.
- No condition implementation is selected through a central switch statement.
- Existing per stage `[[triggers]]` rules produce equivalent compiled trees. The retired global
  `triggers.toml` does not return.
- Explain output identifies the exact failing branch and current progress.

### 2. In game Author Studio

Add an operator only vanilla style editor layered onto the stage map.

Capabilities:

- Create, duplicate, rename, move, connect, and delete stages.
- Drag graph nodes and persist explicit layout coordinates.
- Registry aware pickers for items, blocks, entities, dimensions, tags, recipes, predicates,
  actions, costs, rewards, and enforcement policies.
- A visual condition tree editor for nested boolean logic.
- Trigger, reward, choice, and policy editors.
- Copy, paste, reusable templates, and multi selection edits.
- Undo and redo within an edit session.
- Live validation with file, field, and actionable error details.
- Simulation as a selected player or synthetic test context.
- Read only preview for non operators.
- Reload diff and explicit apply confirmation.

The studio writes deterministic TOML through the same repository used by manual files. It
must preserve unknown fields owned by extensions. Before writing, it creates an atomic backup.

Completion gates:

- Keyboard, mouse, controller, narrator, and GUI scale validation.
- No partial files after a crash or failed validation.
- Unknown extension data survives a load and save cycle.
- An author can build a branching five stage pack without leaving the game.

### 3. Exclusive choices and respec

Support class, faction, research, and mutually exclusive progression paths.

Configuration concepts:

- Named choice groups.
- Minimum and maximum selections.
- Hard and soft conflicts.
- Choice prerequisites and visibility conditions.
- Respec permission, cost, cooldown, command, and script hooks.
- Refund policies for currencies, items, stages, and actions.
- Descendant handling through deny, cascade, suspend, or convert policies.
- Optional irreversible selections with a clear confirmation screen.

Choice mutations run through transactions so an invalid result cannot leave half revoked
branches. FTB Quests integration receives a choice condition and reward action that use the
existing ProgressiveStages stage provider.

Completion gates:

- Team and server scope conflict tests.
- Concurrent purchase and respec tests.
- Complete rollback when any cost, revoke, grant, or reward step fails.
- UI clearly previews every removed and granted stage before confirmation.

### 4. Transactional Java and KubeJS API

Replace loosely related mutations with a transaction builder and structured result.

Required operations:

- Grant, revoke, toggle, purchase, respec, set progress, and change variables.
- Batch mutations against a player, team, or server subject.
- Dry run and explain modes.
- Compare and set revision support for concurrency safety.
- Preflight events that may deny or amend a transaction.
- Post commit events containing the complete immutable change set.
- Failure codes, localized messages, and per operation details.
- Explicit bypass capabilities rather than ambiguous boolean flags.

KubeJS receives the same result model, builders, events, registry access, condition providers,
action providers, transaction hooks, and synchronization controls as the Java API.

Completion gates:

- All commands, GUI purchases, FTB rewards, and scripts use one mutation engine.
- A failed batch makes no persistent changes and emits no post commit event.
- Callbacks cannot observe partially applied state.
- API version and capability discovery are public and documented.

### 5. Per rule enforcement policies

Turn enforcement outcomes into registered, configurable policy chains.

Built in policy identifiers will include:

- Deny the action.
- Warn while allowing the action.
- Hide or mask client presentation.
- Substitute an item, block appearance, loot result, or recipe result.
- Eject a player from a screen, region, or dimension.
- Drop, retain, move, destroy, or downgrade a locked stack.
- Redirect to a position or dimension.
- Run a registered action chain.

Policies declare compatible enforcement contexts and validate their own settings. Pack authors
may set global defaults, category defaults, stage overrides, and individual rule overrides.

Completion gates:

- One shared policy result model is consumed by every enforcer.
- The server remains authoritative even when the client hides content.
- Automation and hopper behavior matches direct player behavior where applicable.
- Invalid policy and context combinations fail reload before replacing live state.

### 6. Complete vanilla standard progression UI

Finish the advancement inspired UI as a first class player experience.

Capabilities:

- Vanilla advancement window, tabs, frames, connectors, backgrounds, and sound language.
- Mouse drag, scroll pan, configurable zoom, keyboard navigation, and controller navigation.
- Remembered category, zoom, pan, filters, and pinned stage per server.
- Search by stage, description, tag, category, locked content, and requirement.
- Hover cards with progress, conditions, costs, rewards, scope, and locked reasons.
- Pinned inspector with dependency path highlighting and breadcrumb navigation.
- Locked, available, owned, temporary, suspended, repeatable, and choice states.
- Optional minimap and dependency legend for large graphs.
- Compare current and proposed states before purchases or respec.
- Narrator labels, focus order, text alternatives, localization, and color blind safe states.

Completion gates:

- Manual validation at small, medium, and 4K GUI scales.
- Graph stress test with at least 500 visible stages.
- No server only class loading on a dedicated client.
- All authoritative actions use request packets and structured server responses.

### 7. Versioned schema, migration, and safe reload

Give every persisted and configured format an explicit schema version.

Required behavior:

- Ordered migrators with idempotence tests.
- Conditional loading by mod, environment, profile, or declared capability.
- Strict required fields and configurable unknown field handling.
- Optional provider warnings instead of crashes.
- Parse, resolve, validate, compile, and index into a candidate snapshot.
- Atomic snapshot swap only after the entire candidate succeeds.
- Timestamped backup and last known good recovery.
- Machine readable JSON Schema generated from provider schemas.
- Deterministic serialization and stable field ordering.
- Reload report with added, removed, changed, migrated, and rejected definitions.

Completion gates:

- Failed reload leaves the previous live snapshot untouched.
- Migration fixtures cover every supported historical format.
- Server data downgrade is refused with a clear recovery message.
- A clean install and each legacy layout generate the documented folder structure.

### 8. Extensible registries and service interfaces

Make namespaced registries the core extension mechanism.

Public registries:

- Condition types.
- Action and reward types.
- Cost and currency types.
- Enforcement contexts and policy types.
- Scope providers.
- Variable types.
- Stage state decorators and tooltip providers.
- Importers, exporters, validators, and compatibility capabilities.

Each entry owns its identifier, codec, validation, optional schema contribution, event interests,
and execution adapter. Registries freeze before the first stage snapshot compiles. Duplicate IDs
and late registration fail clearly.

Compatibility modules discover services by capability, not by core references to third party
classes. Reflection is isolated to version bridges when a dependency has no stable public API.

Completion gates:

- A small test extension adds a condition, action, currency, and tooltip without editing core.
- Missing optional services are reported once and never crash stage loading.
- Registry dumps are available from commands and KubeJS.
- FTB Quests uses the same public capability layer as other integrations.

### 9. Repeatable, seasonal, and prestige stages

Support progression loops without turning stages into a quest replacement.

Modes:

- One time stages.
- Repeatable stages with count, cooldown, and reset policy.
- Seasonal stages with configured windows and grace periods.
- Prestige stages that reset selected progression for permanent benefits.
- Server first or team first achievements.

Progress is delta based and subscribes to relevant events. Anti farming policies include
per entity, per chunk, per source, per time window, and diminishing returns providers.

Completion gates:

- Clock rollback and restart behavior is deterministic.
- Rewards are idempotent across reconnect and crash recovery.
- No arbitrary per tick expression evaluation.
- Seasonal state exposes testable clock providers.

### 10. History, diagnostics, explain, and rollback

Persist an append only mutation journal with configurable retention.

Commands and API:

- Explain why a subject has or lacks a stage.
- Explain why content is locked and identify every requiring stage.
- Show condition progress and the last event that changed it.
- Show grant, revoke, purchase, respec, expiry, and administrative history.
- Validate all loaded definitions and registry references.
- Diff the candidate reload against live state.
- Roll back a selected reversible transaction through the transaction engine.

Sensitive details respect permission levels. Journal writes are buffered, bounded, and resilient
to malformed old entries.

Completion gates:

- Every stage mutation has cause, actor, subject, timestamp, revision, and change set.
- Explain output is stable enough for UI, commands, and scripts to share.
- Rollback is denied when an irreversible action cannot be compensated.

### 11. Declarative action and reward pipeline

Generalize unlock effects and rewards into registered action chains used by grants, revokes,
choices, repeats, enforcement, commands, and integrations.

Pipeline features:

- Ordered sequential and grouped parallel actions.
- Conditions on individual actions.
- Compensation action for transactional rollback where possible.
- Exactly once execution keys for non idempotent rewards.
- Retry and failure policy selected per action.
- Built in item, experience, effect, command, teleport, message, sound, particle, stage,
  variable, loot table, and FTB Quest actions.

Commands and scripts are disabled by default for untrusted imported packs and require explicit
server configuration.

Completion gates:

- Provider actions validate before reload.
- Crash recovery does not duplicate exactly once rewards.
- Irreversible actions are marked and surfaced during dry run and rollback.

### 12. Declared variables, currencies, and formulas

Add typed state that packs can use without inventing scoreboard conventions.

Supported concepts:

- Integer, decimal, boolean, string, duration, and resource identifier variables.
- Player, team, and server scopes.
- Default, minimum, maximum, persistence, visibility, and synchronization controls.
- Registered currency adapters for experience, levels, items, scoreboards, and custom providers.
- Compiled formulas for comparisons and calculated costs.
- Explicit dependency lists so formulas reevaluate only when an input changes.

Completion gates:

- Formula parsing has resource and complexity limits.
- Numeric overflow and invalid conversions return structured errors.
- Secret server variables never synchronize to clients or unprivileged scripts.
- Currency withdrawal participates in the same transaction as the purchase.

### 13. Templates, includes, profiles, and pack composition

Reduce duplication while keeping the final loaded model understandable.

Features:

- Named stage and rule templates.
- Explicit includes with cycle detection.
- Profile activation for difficulty, server type, mod set, or pack mode.
- Parameterized template values with typed defaults.
- Namespace ownership and conflict policy for composed packs.
- A resolve command that exports the fully expanded effective configuration.

Completion gates:

- Resolution order is deterministic and documented.
- Include and inheritance cycles produce a complete path in the error.
- The Author Studio can show whether each value is local, inherited, or overridden.

### 14. Event index, performance budgets, and observability

Build performance into the rule engine rather than adding tick throttles after problems appear.

Features:

- Index compiled conditions by event type, resource target, subject, and scope.
- Dirty only affected subjects and rule plans.
- Coalesce repeated changes within the same server tick.
- Configurable evaluation, synchronization, journal, and integration budgets.
- Metrics for candidate counts, evaluations, cache hits, failures, packet sizes, and queue depth.
- Slow rule diagnostics with stage and provider identifiers.
- Bounded caches with lifecycle cleanup on reload and server stop.

Completion gates:

- A 500 stage stress fixture does not scan all stages for a single item event.
- Reload and disconnect clear every server owned cache.
- Metrics can be read through commands and the Java API without requiring another mod.

### 15. Pack testing, simulation, import, and export toolkit

Make complex progression packs testable before players encounter them.

Features:

- Headless graph and schema validation Gradle task.
- Synthetic player, team, inventory, variable, and event contexts.
- Scenario files with given, when, and expected stage assertions.
- Graph checks for unreachable, contradictory, orphaned, and accidental hidden stages.
- Cost and reward balance reports.
- Stage graph export to JSON and Graphviz.
- Import adapters implemented through the importer registry.
- Compatibility report for configured providers and installed mods.

Completion gates:

- Pack tests run without launching a graphical client.
- CI receives machine readable failures with file and field paths.
- Simulation never writes production stage state.

## Execution phases

### Phase A. Stabilize the current 3.0 baseline

Deliverables:

- Preserve current config migration, merged triggers, UI, commands, KubeJS, and integrations.
- Repair correctness, lifecycle, validation, dedicated client, and build issues found by audit.
- Expand unit tests around dependency modes, parsing contracts, and bulk API behavior.
- Complete the real client and dedicated server release matrix.

Exit gate: the current 3.0 feature set has a clean build and no known release blocking defect.

### Phase B. Extensibility kernel

Implements feature groups 1, 4, 7, and 8.

Deliverables:

- Registry API and freeze lifecycle.
- Versioned schemas and candidate snapshot reload.
- Condition tree compiler with legacy adapters.
- Transaction engine and structured results.
- Public capability and service interfaces.

Exit gate: core conditions and every mutation use the new kernel, and an external test provider
can extend it without a core edit.

### Phase C. Policy and action unification

Implements feature groups 5, 11, and 12.

Deliverables:

- Enforcement policy registry and shared outcomes.
- Declarative action pipelines and compensation.
- Variables, currencies, and compiled formulas.
- Command, GUI, KubeJS, and FTB Quests migration to transactions and actions.

Exit gate: no enforcement category or reward needs a hardcoded parser path.

### Phase D. Player progression experience

Implements feature groups 3, 6, and 9.

Deliverables:

- Completed advancement style UI.
- Exclusive choices and transactional respec.
- Repeatable, seasonal, prestige, and first completion states.

Exit gate: all progression states are visible, accessible, server authoritative, and covered by
multi client tests.

### Phase E. Authoring and pack engineering

Implements feature groups 2, 13, and 15.

Deliverables:

- In game Author Studio.
- Templates, includes, profiles, and resolved export.
- Headless validation, scenarios, simulation, and graph exports.

Exit gate: a pack author can create, validate, simulate, and safely apply a complex progression
pack using only public tools.

### Phase F. Operations and scale

Implements feature groups 10 and 14, with hardening across all earlier groups.

Deliverables:

- Mutation journal, explain engine, diff, and rollback.
- Event indexes, budgets, metrics, cache bounds, and stress tooling.
- Compatibility matrix and long running server tests.

Exit gate: administrators can explain and recover progression changes, and the stress suite meets
the published performance budget.

## FTB Quests integration expansion

The existing integration is retained. Work adds capability based adapters around it:

- ProgressiveStages condition and task types backed by the universal condition engine.
- Stage, choice, variable, currency, and action rewards through transactions.
- Immediate, coalesced quest rechecks after committed progression changes.
- Shared team and server scope resolution.
- Explain information in quest tooltips when the FTB API permits it.
- Capability detection and one warning when an expected FTB API is unavailable.
- Integration tests against supported FTB Quests versions.

ProgressiveStages will not implement quest chapters, narrative pages, general task composition,
or quest reward selection already owned by FTB Quests.

## Compatibility policy

Core may know Minecraft and NeoForge types where required by the platform. Optional mod types
stay inside compatibility modules. Core communicates through namespaced registries, capability
interfaces, immutable request and result records, and events.

An integration is accepted when it:

1. Loads safely when the target mod is absent.
2. Declares the target versions or capability it supports.
3. Registers public providers instead of adding core switch branches.
4. Has a failure mode that leaves core progression usable.
5. Includes a diagnostic status and focused tests or a manual verification recipe.

## Verification matrix

Every phase runs:

- Unit tests for pure models, codecs, validation, compilation, and transaction behavior.
- Migration fixture tests.
- Game tests for server events and enforcement where NeoForge supports them.
- Dedicated server tests with two clients for player, team, and server scope.
- Optional mod matrix for no integrations, each integration alone, and supported combinations.
- UI tests at multiple scales plus manual controller and narrator review.
- Stress packs with large graphs and lock registries.
- Clean clone dependency resolution and `./gradlew clean build`.
- Patch whitespace and repository status review.

## Release and compatibility promises

- Stage files remain under `config/progressivestages/stages`.
- Existing per stage flat trigger rules continue to load through a documented compatibility
  adapter. Triggers remain owned by their stage files.
- Deprecated Java and KubeJS methods remain for at least one minor release and delegate to the
  new transaction API when semantics are safe.
- Migrations never overwrite an existing destination without an explicit conflict report.
- A failed reload never replaces the last valid runtime snapshot.
- Phase work stays on an `envy` branch until approved, then main advances by fast forward and the
  approved commit receives a lightweight phase tag.

## Immediate next work

1. Complete the real client, dedicated server, multiplayer, and optional mod Phase A matrix.
2. Build the registry and schema kernel without changing existing stage semantics.
3. Adapt the current trigger condition types into registry entries.
4. Introduce fully immutable compiled snapshots and atomic registry replacement.
5. Introduce the transaction result model and move one mutation path at a time.
6. Use the FTB Quests bridge as the first external capability integration test.

The architectural constraint for every new implementation is simple. If a pack author or another
mod could reasonably need a different behavior, core exposes a registered provider or configured
policy instead of embedding that choice in a central conditional branch.
